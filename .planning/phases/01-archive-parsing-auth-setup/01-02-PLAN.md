<objective>
Establish X API authentication: environment variable storage, credential verification via `GET /2/users/me`, and documentation of alternatives to the paid X API tier.
</objective>

<context>
The project needs X API credentials. The current research indicates:
- X Basic tier costs $100/month -- this is a significant commitment
- Alternative scraping services (Apify, Bright Data) exist as lower-cost options
- tweepy >= 4.14.0 is the recommended X API client
- Credential verification must happen before any batch operations

Tasks:
1. Build `src/auth/x_auth.py` with credential loading + verification (PARSE-01, AUTH-02)
2. Create `.env.example` template (AUTH-01)
3. Research and document X API alternatives (AUTH-03) -- run in parallel with parser work

Verification endpoint: `GET /2/users/me` using tweepy client.

Required env vars:
- `X_API_KEY` -- API key
- `X_API_SECRET` -- API secret
- `X_ACCESS_TOKEN` -- OAuth access token
- `X_ACCESS_TOKEN_SECRET` -- OAuth access token secret
- `X_BEARER_TOKEN` -- Bearer token (app-only, optional if above provided)

Interface:
```python
@dataclass
class XAuth:
    api_key: str
    api_secret: str
    access_token: str
    access_token_secret: str
    bearer_token: str | None = None

    def verify(self) -> dict:
        """Call GET /2/users/me. Returns user dict on success. Raises AuthError on failure."""

def get_auth() -> XAuth:
    """Load from environment variables. Raises AuthError if required vars missing."""

def verify_credentials(auth: XAuth) -> dict:
    """Verify credentials with GET /2/users/me. Returns user dict or raises AuthError."""
```

For AUTH-03: Research Apify (twitter-followers-scraper) and Bright Data (X scraping API) as alternatives. Document in `docs/auth-alternatives.md` with:
- Pricing (at least approximate)
- What data they can fetch
- How they handle rate limits / anti-detection
- Tradeoffs vs. official X API
</context>
