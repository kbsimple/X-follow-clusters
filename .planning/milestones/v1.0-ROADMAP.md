# Milestone v1.0: MVP

**Status:** ✅ SHIPPED 2026-04-06
**Phases:** 1-6
**Total Plans:** 12

## Overview

MVP milestone delivering a complete end-to-end pipeline: parse X data archive → enrich profiles via X API + scraping → cluster into named groups → review/approve → create native X API lists + export data.

## Phases

### Phase 1: Archive Parsing + Auth Setup

**Goal:** User can load their X data archive and the system is authenticated with X API
**Depends on:** Nothing
**Plans:** 2 plans

Plans:
- [x] 01-01: Archive parsing and auth setup
- [x] 01-02: Auth CLI and verification

**Details:**
JS-wrapped JSON parser (`follower.js`) extracting account IDs and usernames with robust per-entry error handling. X API auth via tweepy OAuth 1.0a with credential verification.

---

### Phase 2: API Enrichment

**Goal:** All followed accounts have rich profile data from the X API
**Depends on:** Phase 1
**Plans:** 1 plan

Plans:
- [x] 02-01: Batch API enrichment with rate limiting, error handling, and immediate caching

**Details:**
867 accounts enriched via `GET /2/users` batch calls (100/req) with exponential backoff (base=1s, max=300s, jitter). Suspended (code 63) and protected (code 179) accounts flagged. All responses cached to `data/enrichment/{account_id}.json` immediately.

---

### Phase 3: Profile Scraping

**Goal:** Supplemental profile data is scraped for fields the X API does not provide
**Depends on:** Phase 2
**Plans:** 1 plan

Plans:
- [x] 03-01: Profile scraping with curl_cffi, BeautifulSoup parser, robots.txt compliance, graceful block handling

**Details:**
curl_cffi-based scraper with TLS impersonation (`impersonate="chrome"`). 7 fields extracted: bio, location, website, join_date, professional_category, pinned_tweet_text, profile_banner_url. Random delays 2-5s with jitter. robots.txt checked via RobotFileParser. Graceful degradation (returns None) when blocked. Reuses ExponentialBackoff from Phase 2.

---

### Phase 4: NLP Clustering

**Goal:** Followers are clustered into meaningful groups using bio text embeddings
**Depends on:** Phase 3
**Plans:** 2 plans

Plans:
- [x] 04-01: Embeddings + semi-supervised K-Means clustering with seed anchoring
- [x] 04-02: LLM-generated descriptive cluster names from member bios

**Details:**
Sentence-transformer embeddings (`all-MiniLM-L6-v2`) for all enriched accounts. Semi-supervised K-Means with 4 seed categories (Geographic, Occupation, Political Action, Entertainment). Cluster size enforced 5-50 members. LLM naming via GPT-4o-mini (OpenAI) or Claude Haiku (Anthropic) with rule-based fallback (8 keyword groups + location detection). Silhouette scoring for quality assessment. CLUSTER-07 and CLUSTER-08 threshold warnings added during verification.

---

### Phase 5: Review Flow

**Goal:** User reviews, approves, and refines clusters before they become lists
**Depends on:** Phase 4
**Plans:** 3 plans

Plans:
- [x] 05-01: Review CLI entry point, approval registry, and cluster size histogram display
- [x] 05-02: Cluster table display with per-member confidence and batch approve
- [x] 05-03: Per-cluster actions (approve/reject/rename/merge/split/defer), merge/split operations, and automation offer

**Details:**
Interactive review CLI (`python -m src.review.cli`) with Rich table display (Name | Members Preview | Size | Silhouette | Status | Category). Per-member silhouette confidence via `sklearn.metrics.silhouette_samples`. Batch approve for size≥10 AND silhouette≥0.5. Per-cluster actions: approve, reject, rename, merge, split, defer. Merge: union re-clustered with k=1. Split: nearest-neighbor reassignment. Automation offer after AUTOMATION_ROUNDS (default=2) new approvals.

---

### Phase 6: List Creation + Export

**Goal:** Approved clusters are created as native X API lists and data is exported
**Depends on:** Phase 5
**Plans:** 3 plans

Plans:
- [x] 06-01: List creation: credential verification, dry-run CLI, HTTP 409 conflict handling, bulk member add, size validation
- [x] 06-02: Data export: Parquet for enriched followers, CSV for approved/deferred clusters
- [x] 06-03: Tests and CLI integration: unit tests for creator and exporter, export wired into --execute mode

**Details:**
Native X API lists created via tweepy `client.create_list()`. Private lists by default. HTTP 409 conflicts pre-checked via `get_owned_lists()`. Members bulk-added in batches of 100 with 0.5s delays. List sizes validated (5-50 per list, <1000 lists per account). Export to Parquet (`data/export/followers.parquet`) and CSV (`data/export/clusters.csv`). 15 unit tests passing.

## Milestone Summary

### Key Decisions

| Decision | Rationale |
|----------|-----------|
| Semi-automated clustering | User wants review/approval before lists are created |
| NLP clustering (not keyword) | Transformer embeddings outperform TF-IDF on short bios |
| Rich profile data (API + scraping) | Maximize information for accurate clustering |
| X API lists as final output | User wants native X app lists, not a separate tool |
| 5-50 people per cluster/list | User-specified; X API hard limit is 5,000 but user prefers smaller lists |
| Coarse granularity (6 phases) | YOLO mode; user wants 6 phases as identified |
| Private lists by default (LIST-D01) | User data is internal; public lists are noisy |
| 0.5s sleep between list creations (LIST-D02) | No official rate limit; conservative delay prevents burst blocks |
| HTTP 409 conflicts pre-checked (LIST-D03) | Avoid partial state; resolve all naming conflicts upfront |
| Parquet for followers, CSV for clusters (EXPORT-D02) | Large machine-readable dataset vs small human-readable output |
| OpenAI preferred over Anthropic (name.py) | Checked first when both API keys present |
