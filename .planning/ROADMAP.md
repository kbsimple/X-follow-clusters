## Phase Details

### Phase 1: Archive Parsing + Auth Setup

**Goal:** User can load their X data archive and the system is authenticated with X API

**Depends on:** Nothing

**Requirements:** PARSE-01, PARSE-02, PARSE-03, AUTH-01, AUTH-02, AUTH-03

**Success Criteria** (what must be TRUE):

1. User points the tool at `data/follower.js` and it extracts all account IDs and usernames without crashing
2. Malformed entries (escaped Unicode, deleted accounts) are logged individually and skipped without halting the run
3. Invalid JSON structure in the archive produces a clear, actionable error message
4. X API credentials are stored in environment variables and verified with `GET /2/users/me` before any data operations
5. If no valid credentials exist, the tool surfaces the auth gap clearly and documents at least one alternative approach (Apify, Bright Data, paid tier)

### Phase 2: API Enrichment

**Goal:** All followed accounts have rich profile data from the X API

**Depends on:** Phase 1

**Requirements:** ENRICH-01, ENRICH-02, ENRICH-03, ENRICH-04, ENRICH-05

**Success Criteria** (what must be TRUE):

1. All follower accounts are enriched via `GET /2/users` batch calls (up to 100 per request) without exceeding rate limits
2. When rate limit headers indicate exhaustion, the system waits with exponential backoff and resumes automatically
3. Suspended accounts (error 63) and protected accounts (error 179) are flagged in the dataset with their status
4. Enriched profiles include bio, location, professional category, pinned tweet text, follower/following counts, and verified status
5. Every API response is cached to disk immediately; no URL is requested twice within a session

### Phase 3: Profile Scraping

**Goal:** Supplemental profile data is scraped for fields the X API does not provide

**Depends on:** Phase 2

**Requirements:** SCRAPE-01, SCRAPE-02, SCRAPE-03, SCRAPE-04, SCRAPE-05

**Success Criteria** (what must be TRUE):

1. Profile pages are scraped for fields not available via API (using curl_cffi for TLS impersonation)
2. Random delays of 2-5 seconds with jitter are applied between scraping requests to avoid blocks
3. `robots.txt` is checked before scraping; scraped fields and legal basis are documented
4. When scraping is blocked, the system falls back gracefully to API-only data without crashing
5. All scraped data is cached locally and never re-requested within a session

**Plans**:
- [x] 03-01-PLAN.md -- Profile scraping with curl_cffi, BeautifulSoup parser, robots.txt compliance, graceful block handling

### Phase 4: NLP Clustering

**Goal:** Followers are clustered into meaningful groups using bio text embeddings

**Depends on:** Phase 3

**Requirements:** CLUSTER-01, CLUSTER-02, CLUSTER-03, CLUSTER-04, CLUSTER-05, CLUSTER-06, CLUSTER-07, CLUSTER-08

**Success Criteria** (what must be TRUE):

1. Bio text is converted to embeddings using `sentence-transformers` (`all-MiniLM-L6-v2`)
2. Clustering is performed on embeddings using HDBSCAN or K-Means (configurable)
3. Every cluster has between 5 and 50 members; clusters violating this are flagged for manual review or re-clustering
4. Cluster names are generated by an LLM based on member profile content, not keyword extraction
5. Clustering is anchored with seed categories (Geographic, Occupation, Political Action, Entertainment) and additional categories are discovered from profile content
6. Cluster size histogram is reported; warning is shown if more than 50% of clusters have fewer than 5 members
7. Silhouette score is computed; clusters with score below 0.3 are flagged as potentially over- or under-clustered

**Plans**:
- [x] 04-01-PLAN.md -- Embeddings + semi-supervised K-Means clustering with seed anchoring
- [x] 04-02-PLAN.md -- LLM-generated descriptive cluster names from member bios

### Phase 5: Review Flow

**Goal:** User reviews, approves, and refines clusters before they become lists

**Depends on:** Phase 4

**Requirements:** REVIEW-01, REVIEW-02, REVIEW-03, REVIEW-04, REVIEW-05, REVIEW-06, REVIEW-07

**Success Criteria** (what must be TRUE):

1. Suggested clusters are displayed grouped by category type with member previews and confidence scores per member
2. User can approve, reject, rename, merge, split, or defer each cluster independently
3. Batch approve is available for clusters with more than a configurable N members and confident names
4. Cluster size distribution is presented before review begins; user is warned if distribution is heavily skewed to small clusters
5. After a configurable number of approved rounds, the system offers to enable full automation mode
6. Deferred clusters do not block approval of other clusters

### Phase 6: List Creation + Export

**Goal:** Approved clusters are created as native X API lists and data is exported

**Depends on:** Phase 5

**Requirements:** LIST-01, LIST-02, LIST-03, LIST-04, LIST-05, EXPORT-01, EXPORT-02

**Success Criteria** (what must be TRUE):

1. Native X API lists are created for all approved clusters with 5-50 members each
2. HTTP 409 naming conflicts are handled gracefully without failing the run
3. Members are bulk-added via `POST /2/lists/{id}/members/add_all` (up to 100 per request)
4. List creation is validated against X limits (5,000 members per list, 1,000 lists per account) and a test call is made before full execution
5. Follower records with enrichment data and cluster assignments are exported to Parquet
6. Approved clusters are exported to CSV with list name, member handles, and cluster metadata
